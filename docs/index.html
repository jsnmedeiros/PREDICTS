PREDICTS Metadata Extraction Pipeline

Auditable extraction from ecological papers into PV07 metadata

Introduction

This repository documents a reproducible pipeline for extracting metadata from ecological papers and preparing the information needed to populate the PREDICTS PV07 form. The pipeline is designed for auditability rather than blind automation: every filled field should be traceable to a source such as the main PDF text, a Docling table, OCR/vision evidence, or downloaded repository data.

The current script is predicts_form_fixed_dryad.py. It was tested here with two Journal of Applied Ecology papers:

  • Jarrett et al. (2021) — bird communities in African cocoa agroforestry.
  • Kingazi et al. (2024) — tropical agroforestry, insect pollinators and bean yield.

These examples are useful because they test different failure modes: Jarrett uses large CSV supplementary files, while Kingazi has a Dryad landing page with one downloadable ZIP that contains the actual CSV files.

Why this pipeline is needed

PREDICTS metadata extraction is difficult because ecological papers rarely report all PV07 fields in one clean table. Important information may be split across the title page, abstract, Methods, maps, table captions, supplementary files, data availability statements and repository ZIP files. A robust workflow therefore needs to combine deterministic rules, repository download logic, table parsing, image evidence and cautious LLM fallback.

Pipeline overview

Evidence sources

Source Purpose Typical PV07 fields supported
Main PDF text Title, abstract, DOI, study area, sampling design, dates, methods Citation, country, region, dates, study description, sampling method
Docling tables Tables embedded in the article PDF Study sites, species/group counts, model/table evidence, sampling units
OCR Text embedded in page images, maps, figure panels and scanned tables Coordinates, site labels, table headings, captions
Vision model Optional interpretation of maps, plots and visual tables Coordinate ranges, site counts, figure-derived metadata
Repository downloads CSV, Excel, ZIP, TXT, JSON, PDF and README files Site counts, species counts, coordinates, years, abundance units
LLM fallback Only fills missing or weak fields after deterministic extraction Ambiguous methods, cleaned descriptions, notes

Execution stages

The script prints an eleven-step audit trail:

  1. Extract PDF text with pdfplumber and write page-aware text evidence.
  2. Write DOI/citation metadata JSON.
  3. Extract real PDF tables with Docling from result.document.tables.
  4. Discover supplementary/data links from explicit URLs, DOI contexts, repository landing pages and Crossref related links.
  5. Download and unpack tabular data, including nested ZIP files.
  6. Export useful PDF pages as images for maps, figures and scanned/visual tables.
  7. Run OCR on selected image pages.
  8. Optionally run a local Ollama vision model on maps/plots/visual tables.
  9. Combine deterministic extraction with optional LLM fallback for missing/weak fields only.
  10. Save a master metadata JSON and an evidence workbook.
  11. Fill the PREDICTS PV07 AcroForm.

Script architecture

The pipeline is modular. The main functional groups are:

Module area Main functions Role
PDF text extract_pdf_pages, full_text_from_pages, section_text Extract and clean page-aware article text.
Tables export_docling_tables Export only real Docling result.document.tables.
Images export_useful_page_images Select pages likely to contain maps, plots, figures or tables.
Vision analyse_page_images_with_vision, ollama_vision_json Create auditable JSON/CSV from visual evidence.
Repository discovery find_links_and_dois, doi_urls_from_data_availability_context, scrape_supplementary_links Find data availability links and repository DOIs.
Repository-specific download dryad_urls_from_identifier, _dryad_file_links_from_landing, figshare_api_urls_from_identifier, zenodo_urls_from_identifier, eidc_urls_from_identifier, osf_urls_from_identifier Convert landing pages/DOIs into direct file downloads.
Data parsing summarize_downloaded_tables, ZIP extraction helpers Parse downloaded CSV/XLSX/ZIP contents and summarize evidence.
Metadata extraction deterministic rules plus ollama_llm_fallback Populate PV07 metadata fields with evidence.
Form filling PDF field mapping and visible form fill Write the final PV07 PDF.

Repository download logic

The repository downloader is deliberately redundant because ecological papers use different repository platforms and publisher PDFs often break URLs/DOIs across lines.

Supported sources

Repository/source Handling strategy
Dryad Detect 10.5061/dryad.*, open Dryad landing/API pages, identify direct file-stream/download links, prioritise ZIP files when present.
Figshare Use Figshare API from article DOI or landing page identifier.
Zenodo Use records API and /files/ URLs.
EIDC/CEH Resolve UUID-style 10.5285/* data package links.
OSF Scrape storage/download URLs.
Publisher supplementary material Scrape article HTML for supplement/mediaobject/download anchors.
Crossref Search related links for datasets or supplementary materials.

Dryad fix tested by Kingazi

Kingazi et al. has a Dryad data availability link:

https://datadryad.org/dataset/doi:10.5061/dryad.3tx95x6pd

The landing page contains a single ZIP, Data_Kingazi_et_al..zip. Inside that ZIP are the usable data files:

  • Bean yield data.csv
  • Insect pollinator data.csv
  • NMDS_Data.csv

The downloader must therefore not stop at the landing page and must not treat random reference links as the data source. It needs to resolve the Dryad DOI to a real file download, save the ZIP, unzip it, and parse the CSVs inside.

Quality-control notes

Issue Why it matters Recommended action
Final metadata says number_of_species = 4, but CaptureDataset.csv contains 515 unique species and BirdCharacteristics.csv contains 259 unique species. This indicates that the final field was probably overwritten or extracted from a narrower phrase/table rather than the repository data. Prefer repository-derived unique species counts when the downloaded capture/species table is present. Flag conflicts in evidence_used_for_form.xlsx.
Final metadata says country = Multiple countries, while repository coordinate data spans Cameroon and Equatorial Guinea. PV07 may require explicit country names or separate study records depending on how the form is completed. Store Cameroon; Equatorial Guinea in comments or split studies if needed.
Site counts differ by source: abstract/methods mention 83 sites; CocoaFarmTreeCover.csv has 38 farms; CaptureDataset.csv has 177 unique site values. Different files represent different sampling units or historical datasets. Keep the article-reported 83 sites when it is the study design number, but record downloaded-file counts as audit evidence.

Main PDF table evidence

The article PDF contained three Docling tables. The exported table samples include model outputs for pollinator abundance/species richness/visitation rate, floral and plant resources, and bean yield/pollination treatments. These are not always direct PV07 fields, but they are useful audit evidence and can support metric/unit interpretation.

Image/map evidence

Page 3 contains the study map and example plot photographs. This is particularly useful for location context and for checking whether the extraction should treat the study as a single study area or multiple plots.

Page 4 contains methods text about the paired 50 × 50 m plots, the March–May 2022 monitoring period, and sampling details.

Quality-control notes

Issue Why it matters Recommended action
Final metadata says number_of_sites = 2, but the Methods report 16 paired agroforestry/non-agroforestry plots, i.e. 32 plots, and the downloaded data uses 32 plot IDs. 2 likely came from the two treatment types rather than the number of sites/plots. Add a rule that recognises paired plot designs: 16 pairs × 2 treatments = 32 plots/sites, while recording treatment count separately.
The paper uses pollinator visits, species richness and visitation rates, but PV07 needs abundance metric/unit and sampling effort unit. These are related but not identical. Use abundance, individuals, and plot/observation effort consistently, with comments on the 10-min observations and five 1 m² subplots.
The Dryad landing page contains a ZIP, and the ZIP contains another ZIP plus README. A shallow downloader will report zero usable supplementary files. Always recursively unzip archives and parse CSVs inside nested ZIP files.

Suggested validation rules before filling PV07

Rule Rationale
Prefer explicit Methods design counts over generic counts found in result tables. Model tables often contain treatment levels, not sites.
Prefer downloaded repository coordinates over map OCR when coordinate columns exist. Repository CSV coordinates are usually more precise and machine-readable.
Do not overwrite article-reported site count with unique values from unrelated supplementary tables unless the table is clearly the sampling-unit table. Unique site IDs can represent visits, farms, mist-net stations, species records, or historical datasets.
Keep a conflict log when PDF text, Docling tables and repository data disagree. Human review is essential for PREDICTS.
Recursively unzip archives and parse nested CSV/XLSX files. Many repositories package data in nested folders or ZIPs.
Treat LLM output as fallback only and validate against PV07 allowed options. Prevents hallucinated fields and invalid dropdown values.
Store both the final field value and the evidence source used. Makes the filled PDF auditable.

Recommended next improvements

  1. Add a qc_flags section to master_metadata.json for conflicts such as species count, site count and country naming.
  2. Add a site_count_source field so the final number can be traced to Methods, repository data, or map evidence.
  3. Add a species_count_source field and prioritise repository species columns when available.
  4. Add treatment-aware logic for paired designs so that two treatments are not misread as two sites.
  5. Add a small HTML summary page per paper with the final metadata table, downloaded files table, extracted evidence images and QC flags.
  6. Add unit tests for repository download functions using known URLs: Dryad, Figshare, Zenodo, EIDC and OSF.

Minimal command-line example

find ../Papers2Test -iname "*.pdf" | while read paper; do
  python predicts_form_fixed_dryad.py     --paper "$paper"     --form PREDICTS_Form_v07.pdf     --compiler-first Joana     --compiler-last Medeiros     --compiler-id JM1     --compiler-email j.medeiros@pgr.reading.ac.uk     --download-dir ./downloaded_data     --model gemma3:4b     --vision-model moondream     --vision-max-pages 15     --max-ocr-pages 15
 done

What to show on the GitHub Pages site

A useful GitHub Pages version should show both the final outputs and the evidence trail:

  • pipeline diagram or stage list;
  • one table summarising all output files;
  • one table per example with final PV07 fields;
  • one table per example with downloaded data files and row/column summaries;
  • page images for maps/figures used as evidence;
  • QC flags showing where the script still needs human review;
  • links to the filled PDF, metadata JSON and evidence workbook.
