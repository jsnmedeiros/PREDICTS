PREDICTS Metadata Extraction Pipeline

Audit-first extraction from ecological papers into PV07 metadata

What this page shows

This page documents the current prototype pipeline for extracting metadata from ecological papers and preparing PREDICTS (Projecting Responses of Ecological Diversity in Changing Terrestrial Systems) PV07 forms.

The main aim is auditability: each filled field should be traceable to evidence from the paper text, Methods section, tables, maps, OCR/vision output, or downloaded repository data.

This is not presented as fully automatic yet. It is an audit-first workflow where difficult fields such as site counts, species counts, dates, coordinates and study splitting are flagged for human validation.

The current workflow demonstrates:

  • PDF text extraction;
  • DOI and citation extraction;
  • Docling table extraction;
  • OCR and optional vision-model evidence;
  • repository download handling, including Dryad ZIPs;
  • nested archive extraction;
  • metadata JSON production;
  • filled PREDICTS PV07 PDF form generation.

The main remaining work is quality control: the pipeline must still be checked for site counts, species counts, dates, coordinates, country names and whether a paper should be split into multiple PV07 studies.

Why this pipeline is needed

Ecological papers rarely report all PV07 fields in one clean table. Important information is often split across:

  • the title page;
  • abstract;
  • Methods and study-area sections;
  • maps and figure captions;
  • model/result tables;
  • supplementary files;
  • data availability statements;
  • README files inside downloaded archives.

A robust workflow therefore needs to combine deterministic extraction rules, repository download logic, table parsing, OCR/vision evidence and cautious LLM fallback. The LLM is used only for missing or weak fields and should not overwrite stronger deterministic evidence.

Pipeline at a glance

PDF paper → extract text and DOI/citation metadata → extract Docling tables → discover supplementary/data links → download files and recursively unzip archives → parse CSV/XLSX/TXT/JSON/PDF evidence → export useful pages as images → OCR and optional vision analysis → deterministic metadata extraction → optional LLM fallback for missing/weak fields → save master JSON and evidence workbook → fill the PREDICTS PV07 PDF form

Evidence sources

Main PDF text Highest-value source for study design, Methods, dates and location descriptions title, abstract, country, study description, sampling method, dates

Repository download logic

The downloader is deliberately redundant because publisher PDFs often break URLs or repository DOIs across lines.

Dryad Figshare Zenodo EIDC/CEH OSF publisher supplements Crossref related links

For Dryad, the key requirement is to avoid stopping at the landing page. The code must resolve the dataset page to actual file downloads, save ZIP files, recursively unzip them, and parse the CSV/XLSX files inside.

Example 2: Kingazi et al. 2024

Why this example matters

Kingazi tests two common failure modes: a Dryad page with a downloadable ZIP, and a paired sampling design that can be misread as either one broad study area or two treatments.

Extracted metadata summary

Field Extracted value
First author Kingazi
Year 2024
Title Tropical agroforestry supports insect pollinators and improves bean yield
Journal Journal of Applied Ecology
DOI 10.1111/1365-2664.14629
Country Tanzania
Study area Eastern lower slopes of Mount Kilimanjaro; Kilimanjaro Region; Northern Tanzania; Chagga home gardens
Sampling method Fixed plots/quadrats
Site count used in filled form 16 paired sampling sites/replicates
Plot-level repository evidence 32 plot IDs in pollinator/NMDS files
Study dates March–May 2022
Number of species 45

Dryad archive evidence

Kingazi has a Dryad data page with one ZIP. Inside that ZIP are the usable CSV files:

  • Bean yield data.csv
  • Insect pollinator data.csv
  • NMDS_Data.csv

The downloader must save the ZIP, unzip it, and parse the CSVs inside. A shallow downloader will incorrectly report zero usable data files.

Image/map evidence

Page 3 contains the study map and example plot photographs. This is particularly useful for location context and for checking whether the extraction should treat the study as a single study area or multiple plots.

Page 4 contains methods text about the paired 50 × 50 m plots, the March–May 2022 monitoring period, and sampling details.

Important site-count interpretation

For the PV07 form, this example currently records 16 sites/paired sampling replicates, because the article describes 16 agroforestry plots paired with 16 non-agroforestry plots. The downloaded data also contains 32 plot-level records, representing the two treatment plots within the 16 paired design.

This distinction is important: the pipeline should not mistake the two treatment classes for two sites, and it should not collapse the whole study map into one site.

Validation rules before filling PV07

Rule Rationale
Prefer explicit Methods design counts over generic counts found in result tables Model tables often contain treatments, not sites
Prefer repository coordinates over map OCR when coordinate columns exist CSV coordinates are usually more precise
Do not overwrite article-reported site counts with unrelated unique IDs unless the table is clearly the sampling-unit table Unique IDs can represent visits, plots, farms, stations or records
Keep a conflict log when PDF text, tables and repository data disagree Human review is essential for PREDICTS
Recursively unzip archives Many repositories package files in nested folders or ZIPs
Treat LLM output as fallback only Prevents hallucinated fields and invalid dropdown values
Store final field value plus source evidence Makes the filled PDF auditable

  4. Add unit tests for repository download functions using known Dryad, Figshare, Zenodo, EIDC and OSF examples.

Minimal command-line example

find ../Papers2Test -iname "*.pdf" | while read paper; do
  python predicts_form.py \
    --paper "$paper" \
    --form PREDICTS_Form_v07.pdf \
    --compiler-first Joana \
    --compiler-last Medeiros \
    --compiler-id JM1 \
    --compiler-email j.medeiros@pgr.reading.ac.uk \
    --download-dir ./downloaded_data \
    --model gemma3:4b \
    --vision-model moondream \
    --vision-max-pages 15 \
    --max-ocr-pages 15
done
